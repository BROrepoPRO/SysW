## 1. Анализ текущей структуры `.ycarules`

Файл `.ycarules` (117 строк) имеет следующую структуру:

| Раздел | Номер | Правила | Описание |
|--------|-------|---------|----------|
| Легенда | — | — | Префиксы `[Gx]`, `[Zx]`, `[Kx]`, `[Ux]`, `[Sx]` и механизм отключения через `<!-- ... -->` |
| Распределение ролей | 1 | `[O1]` | Orchestrator и специализация агентов |

| Запреты и ограничения | 3 | `[Z1]`–`[Z6]` | Ограничения на импорт, удаление, кириллицу, правки |

**Стиль оформления:**
- Заголовки разделов: `## N. Название`
- Правила: `[ПрефиксN] Текст правила`

- Комментарии отключения: `<!-- [Xx] ... -->`
- Кодировка: UTF-8

## 2. Предлагаемая структура новых разделов

### Раздел 7: Автодополнение текста (Text Autocompletion)

**Префикс:** `[Tx]` — Text completion

| ID | Название | Суть | На что влияет |
|----|----------|------|---------------|
| `[T1]` | Автодополнение в VBA-комментариях | При написании комментариев на русском языке (`' текст`) предлагать завершение фраз | Поведение ассистента при редактировании `.bas`/`.cls`/`.frm` файлов |
| `[T2]` | Автодополнение в Markdown-документации | При написании `.md`-файлов на русском языке предлагать завершение предложений | Поведение ассистента при работе с документацией |
| `[T3]` | Автодополнение в текстовых файлах | При написании любых текстовых файлов проекта на русском языке предлагать автодополнение | Поведение ассистента при работе с любыми текстовыми файлами |
| `[T4]` | Контекстное автодополнение | Учитывать терминологию проекта при автодополнении | Качество предлагаемых завершений (чтобы ассистент использовал правильные имена модулей и термины) |

### Раздел 8: Система правил исключений и обязательного вхождения (Exclude/Include)

**Префикс:** `[Ex]` — Exclude/Include

| ID | Название | Суть | На что влияет |
|----|----------|------|---------------|
| `[E1]` | Исключение файлов/папок по glob | Ассистент не читает, не анализирует и не изменяет указанные файлы/папки | Какие файлы ассистент **полностью игнорирует** (экономит контекст, предотвращает случайные правки) |
| `[E2]` | Исключение участков кода | Участки кода с маркером `' SKIP_ASSIST` не предлагаются к изменению | Какие участки кода ассистент **не трогает** (защита сгенерированного/легаси кода) |
| `[E3]` | Обязательное вхождение файлов/папок | Указанные файлы/папки всегда учитываются в контексте | Какие файлы ассистент **всегда держит в уме** (даже если не открыты) |
| `[E4]` | Обязательное вхождение модулей/функций при генерации кода | Указанные модули/функции всегда в контексте при генерации кода | Какие API ассистент **обязательно учитывает** при написании нового кода |
| `[E5]` | Приоритет правил | Include переопределяет Exclude для пересекающихся паттернов | Разрешение конфликтов между правилами |

```ycarules
## 7. Автодополнение текста (Text Autocompletion)

<!--
  Раздел управляет автодополнением текста на русском языке.
  Влияет на: поведение ассистента при написании комментариев,
  документации и любых текстов в проекте.
  Чтобы отключить правило, оберните его в <!-- ... -->.
-->

[T1] **Автодополнение в VBA-комментариях**
При написании комментариев на русском языке (строки, начинающиеся с `'`) активно предлагать завершение фраз и предложений. Учитывать стиль существующих комментариев в проекте: краткие описания функций, пояснения параметров, секции-разделители.
Влияет на: файлы `.bas`, `.cls`, `.frm`.

[T2] **Автодополнение в Markdown-документации**
При написании Markdown-файлов (`.md`) на русском языке активно предлагать завершение предложений. Учитывать:
- Структуру документа (заголовки, списки, таблицы)
- Стиль документации проекта (технический, с примерами кода)
- Терминологию проекта (названия модулей, функций, скриптов)
Влияет на: файлы `.md`.

[T3] **Автодополнение в текстовых файлах**
При написании любых текстовых файлов проекта (`.txt`, конфиги, readme) на русском языке предлагать автодополнение фраз. Для файлов без явной структуры — опираться на общий контекст проекта и соседние строки.
Влияет на: `.txt`, `.cfg`, `.ini`, `README.md` и прочие текстовые форматы.

[T4] **Контекстное автодополнение**
При автодополнении текста на русском языке учитывать:
- Названия модулей проекта (`Mod_*`, `Sheet_*`)
- Имена функций и процедур
- Ключевые термины предметной области (заказ-наряд, импорт, SQLite, шапка заказа)
- Принятые сокращения и аббревиатуры
Влияет на: качество и релевантность предлагаемых завершений.

## 8. Система правил исключений и обязательного вхождения (Exclude/Include)

<!--
  Раздел определяет, какие файлы, папки и участки кода должны
  обрабатываться ассистентами, а какие — игнорироваться.
  Поддерживаются glob-паттерны: * (любые символы), ** (любая вложенность),
  ? (один символ).
  Влияет на: какие файлы ассистент видит, читает и может изменять.
  Чтобы отключить правило, оберните его в <!-- ... -->.
-->

[E1] **Исключение файлов и папок по glob-паттернам**
Ассистент НЕ должен читать, анализировать или изменять файлы и папки,
соответствующие указанным паттернам. Паттерны указываются в кавычках,
через запятую.
Влияет на: экономию контекстного окна, предотвращение случайных правок.

Пример:
  Exclude: "*.log", "_temp_export/*", "backup/", "*.tmp"

Пользовательские исключения (добавьте свои паттерны ниже):
  Exclude: ""

[E2] **Исключение участков кода по маркеру SKIP_ASSIST**
Участки кода, содержащие маркер `' SKIP_ASSIST` в VBA-файлах или
`<!-- SKIP_ASSIST -->` в Markdown/HTML, не должны предлагаться
ассистентом к изменению. Ассистент может читать такие участки
для понимания контекста, но не должен их модифицировать.
Влияет на: защиту критических или автоматически сгенерированных блоков кода.

Пример VBA:
  ' SKIP_ASSIST — этот блок сгенерирован внешним инструментом
  Public Function LegacyCalc() As Integer
      ' ... сохранённый код ...
  End Function

Пример Markdown:
  <!-- SKIP_ASSIST -->
  Секция сгенерирована автоматически, не редактировать.

[E3] **Обязательное вхождение файлов и папок (Include)**
Указанные файлы и папки ассистент ОБЯЗАН учитывать при анализе
контекста, даже если они не открыты в редакторе. Паттерны
указываются в кавычках, через запятую.
Влияет на: полноту контекста — ассистент всегда «помнит» о ключевых файлах.

Пример:
  Include: "docs/*.md", "src/modules/Mod_Utils.bas",
           "src/modules/Mod_Constants.bas"

Рекомендуемые для включения (редактируется пользователем):
  Include: "docs/DEVELOPER.md", "docs/ARCHITECTURE_SQLITE.md",
           "src/modules/Mod_Constants.bas"

[E4] **Обязательное вхождение модулей и функций при генерации кода (IncludeContext)**
При генерации нового кода или изменении существующего ассистент
ОБЯЗАН учитывать сигнатуры и логику указанных модулей и функций.
Это предотвращает дублирование и конфликты имён.
Влияет на: качество генерируемого кода — ассистент не создаёт
дублирующие функции и использует существующие API.

Пример:
  IncludeContext: "Mod_Utils.*", "Mod_Logger.WriteLog",
                  "Mod_Constants.*"

Пользовательские настройки (добавьте свои модули ниже):
  IncludeContext: ""

[E5] **Приоритет правил**
При пересечении паттернов Exclude и Include действует следующий
приоритет:
1. Явное Include переопределяет Exclude для конкретного файла/папки
2. Более конкретный паттерн имеет приоритет над менее конкретным
   (например, "src/modules/Mod_Utils.bas" важнее "src/modules/*")
3. При равной специфичности приоритет у Include
Влияет на: разрешение конфликтов между правилами.
```

## 4. Обоснование решений

### Выбор префиксов
- `[Tx]` — следует логике существующих префиксов: `T` = Text completion, не пересекается с существующими `G`, `Z`, `K`, `U`, `S`.
- `[Ex]` — `E` = Exclude/Include, не пересекается с существующими.

### Механизм отключения правил
Полностью соответствует существующему: `<!-- [E1] ... -->` — обернуть правило в комментарий HTML-разметки.

### Glob-паттерны
Используются стандартные glob-паттерны, знакомые пользователям по `.gitignore`:
- `*` — любая последовательность символов (кроме `/`)
- `**` — любая последовательность символов, включая `/`
- `?` — один любой символ

### IncludeContext vs Include
Разделение на `[E3]` (файлы для контекста) и `[E4]` (модули/функции для генерации кода) позволяет точнее управлять поведением ассистента: одно дело — знать о существовании файла, другое — обязательно учитывать его API при генерации нового кода.

### Встроенные исключения
Системные исключения (`.codeassistantignore`) не дублируются в `.ycarules`. Пользователь добавляет только свои паттерны в `[E1]`.

## 5. Сводная таблица влияния правил на работу ассистента

| Что делает ассистент | Какие правила на это влияют | Эффект |
|---------------------|----------------------------|--------|
| Пишет комментарий в VBA | `[T1]`, `[T4]` | Автодополнение фраз на русском с учётом терминов проекта |
| Пишет документацию в .md | `[T2]`, `[T4]` | Автодополнение с учётом структуры документа и терминологии |
| Редактирует любой текст | `[T3]`, `[T4]` | Автодополнение на русском в любых текстовых файлах |
| Читает файлы проекта | `[E1]`, `[E3]` | Исключённые файлы не читаются; включённые — всегда в контексте |
| Предлагает изменения в коде | `[E2]`, `[E4]` | Блоки с `SKIP_ASSIST` не трогает; указанные модули учитывает |
| Генерирует новый код | `[E4]` | Использует существующие API из IncludeContext, не дублирует |
| Конфликт правил | `[E5]` | Include важнее Exclude при пересечении |
