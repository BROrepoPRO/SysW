## Детальный список проблем

### 1. Mod_OrderHeader.bas

#### OH-01: Отсутствие проверки `Nothing` для `wsModel` (строка 39) — **Critical**
- **Файл:** [`src/modules/Mod_OrderHeader.bas`](src/modules/Mod_OrderHeader.bas:39)
- **Описание:** В функции `FillHeaderFromOrder` лист `wsModel` получается через прямое обращение `ThisWorkbook.Sheets("model")`. Если лист `model` отсутствует, VBA выбросит необработанную ошибку. При этом `wsSpisok` и `wsMain` также не проверяются.
- **Рекомендация:** Использовать `GetSheetByName(ThisWorkbook, "model")` с проверкой на `Nothing` для всех трёх листов, аналогично тому, как это сделано в `FindOrder`.

#### OH-02: Непроверенное значение `ModelCode` при поиске на листе model (строка 79) — **High**
- **Файл:** [`src/modules/Mod_OrderHeader.bas`](src/modules/Mod_OrderHeader.bas:79)
- **Описание:** `lastModelRow` вычисляется через `wsModel.Cells(wsModel.Rows.Count, 1).End(xlUp).Row`. Если лист `model` пуст, `lastModelRow` будет равен 1, и диапазон `A3:A1` вызовет ошибку.
- **Рекомендация:** Проверить, что `lastModelRow >= 3` перед формированием диапазона поиска.

#### OH-03: Неверный вызов `Mod_Utils.WriteLog` в обработчике ошибок (строка 202) — **Medium**
- **Файл:** [`src/modules/Mod_OrderHeader.bas`](src/modules/Mod_OrderHeader.bas:202)
- **Описание:** В `FindOrder_UI` в блоке `ErrHandler` используется `Mod_Utils.WriteLog("FindOrder_UI: " & Err.Description)`, но `Mod_Utils.WriteLog` принимает только один параметр (message). При этом в `FillHeaderFromOrder_UI` (строка 159) корректно используется `Mod_Logger.WriteLog` с двумя параметрами. Несоответствие вызовов.
- **Рекомендация:** Заменить на `Mod_Logger.WriteLog("Mod_OrderHeader", "FindOrder_UI: " & Err.Description)`.

#### OH-04: `FillHeaderFromOrder` не восстанавливает состояние при ошибке (строка 42) — **Low**
- **Файл:** [`src/modules/Mod_OrderHeader.bas`](src/modules/Mod_OrderHeader.bas:42)
- **Описание:** Функция очищает `B3:B15` в начале работы. При ошибке после очистки (например, на строке 64-70 при копировании данных) диапазон останется пустым, хотя исходные данные могли быть утеряны безвозвратно.
- **Рекомендация:** Сохранять состояние `B3:B15` перед очисткой и восстанавливать при ошибке.

### 2. Mod_Import.bas

#### IM-01: `ImportSheet` — отсутствие обработки ошибок (строка 17) — **Critical**
- **Файл:** [`src/modules/Mod_Import.bas`](src/modules/Mod_Import.bas:17)
- **Описание:** Процедура `ImportSheet` не имеет блока `On Error`. Любая ошибка (например, `wsSource.Copy` при проблемах с книгой) приведёт к аварийному завершению. При этом `wsSource` получен из внешней книги `report.xlsx`, которая может быть повреждена.
- **Рекомендация:** Добавить `On Error GoTo ErrHandler` с корректным восстановлением состояния и сообщением пользователю.

#### IM-02: `ImportSheet` — не проверяется успешность переименования листа (строка 33-35) — **High**
- **Файл:** [`src/modules/Mod_Import.bas`](src/modules/Mod_Import.bas:33)
- **Описание:** После копирования листа имя задаётся через `ActiveSheet.Name = newName` с `On Error Resume Next`. Если `newName` содержит недопустимые символы (длинное значение из B2), ошибка проглатывается, и лист остаётся с именем по умолчанию.
- **Рекомендация:** Проверить `newName` на длину (макс. 31 символ для Excel) и недопустимые символы перед присвоением.

#### IM-03: `ImportDataToMain` — не проверяется `Nothing` для `wsSource` (строка 45) — **High**
- **Файл:** [`src/modules/Mod_Import.bas`](src/modules/Mod_Import.bas:45)
- **Описание:** Функция `ImportDataToMain` принимает `wsSource As Worksheet`, но не проверяет его на `Nothing`. Вызов из `ImportDataToMain_UI` (строка 198) передаёт `ActiveSheet`, который может быть `Nothing` в редких случаях, но вызов из `ImportSheet` (строка 37) гарантирует наличие. Однако прямая передача `Nothing` из другого кода вызовет ошибку.
- **Рекомендация:** Добавить проверку `If wsSource Is Nothing Then Exit Sub` в начале процедуры.

#### IM-04: `ImportDataToMain` — граничные значения `lastRow` (строка 61) — **Medium**
- **Файл:** [`src/modules/Mod_Import.bas`](src/modules/Mod_Import.bas:61)
- **Описание:** `lastRow` вычисляется через `Max(lastRowL, lastRowX)`. Если оба столбца пусты, `lastRow` будет равен 1 (или 2 после проверки). Очистка `Range("L2:N" & lastRow)` при `lastRow = 2` очистит только строку 2, что корректно. Но если `lastRow = 1`, то `Range("L2:N1")` — некорректный диапазон.
- **Рекомендация:** Убедиться, что при `lastRow < 2` очистка не выполняется, или использовать `If lastRow >= 2 Then ...`.

#### IM-05: `ImportSheet_UI` — не проверяется значение B4 (строка 128) — **Low**
- **Файл:** [`src/modules/Mod_Import.bas`](src/modules/Mod_Import.bas:128)
- **Описание:** `ImportSheet_UI` читает ГРЗ из ячейки B4 без проверки на пустое значение. Если B4 пуста, `ImportSheet` получит пустую строку, `ExtractNumberFromGRZ` вернёт `""`, и `SearchSheetByGRZ` завершится с сообщением "Лист с ГРЗ не найден", что может сбить пользователя с толку.
- **Рекомендация:** Добавить проверку `IsEmpty` или `""` для B4 перед вызовом `ImportSheet`.

### 4. Mod_SheetOps.bas

#### SO-01: `SearchSheetByGRZ` — утечка объекта Workbook при ошибке (строка 67-68) — **Critical**
- **Файл:** [`src/modules/Mod_SheetOps.bas`](src/modules/Mod_SheetOps.bas:67)
- **Описание:** Книга `report.xlsx` открывается с `On Error Resume Next`. Если открытие не удалось, `wbReport` остаётся `Nothing`, что корректно обрабатывается. Однако если книга открылась, но в процессе поиска листа возникла ошибка (например, повреждённый лист), управление перейдёт на метку `CleanUp`, где книга будет закрыта. **НО:** если ошибка возникла ДО присвоения `wbReport` (на строке 67), `wbReport` может быть в некорректном состоянии. Кроме того, при ошибке внутри цикла `For Each ws In wbReport.Sheets` (строка 78) — `wbReport` будет закрыт, но `wsResult` может указывать на лист из закрытой книги.
- **Рекомендация:** Усилить обработку ошибок: использовать флаг для отслеживания состояния открытия книги, обнулять `wsResult` при ошибке.

#### SO-02: `RenameSheetsByGRZ` — удаление листа без проверки (строка 123) — **Critical**
- **Файл:** [`src/modules/Mod_SheetOps.bas`](src/modules/Mod_SheetOps.bas:123)
- **Описание:** `existingWs.Delete` вызывается без предупреждения. Если `grzNumber` совпадёт с именем существующего листа в `report.xlsx`, лист будет безвозвратно удалён. В VBA удаление листа не отменяется (undo). Это может привести к потере данных.
- **Рекомендация:** Добавить подтверждение пользователя перед удалением листа, либо переименовывать существующий лист, а не удалять.

#### SO-03: `RenameSheetsByGRZ` — отсутствие обработки ошибок (строка 96) — **High**
- **Файл:** [`src/modules/Mod_SheetOps.bas`](src/modules/Mod_SheetOps.bas:96)
- **Описание:** Вся процедура `RenameSheetsByGRZ` не имеет блока `On Error GoTo`. Используется только `On Error Resume Next` для отдельных операций. Любая необработанная ошибка (например, при сохранении `wbReport.Save` на строке 133) приведёт к аварийному завершению.
- **Рекомендация:** Добавить общий обработчик `On Error GoTo ErrHandler` с закрытием книги и сообщением пользователю.

#### SO-04: `ClearMainSheet_UI` — `UsedRange.Rows.Count` может быть неточным (строка 157) — **Medium**
- **Файл:** [`src/modules/Mod_SheetOps.bas`](src/modules/Mod_SheetOps.bas:157)
- **Описание:** `UsedRange.Rows.Count` в Excel может возвращать некорректное значение, если лист содержит форматирование за пределами данных. Это может привести к очистке слишком большого диапазона или, наоборот, неполной очистке.
- **Рекомендация:** Использовать `Cells.Find("*", , , , xlByRows, xlPrevious).Row` для более точного определения последней строки.

#### SO-05: `ExtractNumberFromGRZ` — нет защиты от очень длинных строк (строка 19) — **Low**
- **Файл:** [`src/modules/Mod_SheetOps.bas`](src/modules/Mod_SheetOps.bas:19)
- **Описание:** Функция не имеет ограничения на длину входной строки. При передаче строки длиной > 1000 символов цикл будет выполняться долго, хотя на практике ГРЗ редко превышает 15 символов.
- **Рекомендация:** Добавить проверку `If Len(grz) > 50 Then grz = Left(grz, 50)` или аналогичное ограничение.

### 5. Sheet1_main.cls

#### SM-01: `Worksheet_Change` — рекурсивный вызов через `FillHeaderFromOrder` (строка 26) — **High**
- **Файл:** [`src/sheets/Sheet1_main.cls`](src/sheets/Sheet1_main.cls:26)
- **Описание:** Событие `Worksheet_Change` использует `Static isProcessing As Boolean` для предотвращения рекурсии. Однако `FillHeaderFromOrder` (строка 26) может изменять ячейки на листе `main` (B3:B15), что вызовет повторное событие `Worksheet_Change`. Флаг `isProcessing` защищает от рекурсии, но если `FillHeaderFromOrder` вызовет ошибку, флаг будет сброшен в `CleanUp` (строка 30). **НО:** если ошибка произойдёт ДО установки `isProcessing = True` (строка 18), флаг не будет установлен, и рекурсия возможна.
- **Рекомендация:** Устанавливать `isProcessing = True` сразу после объявления переменных, до любых операций с листом.

#### SM-02: `Worksheet_Change` — `On Error GoTo CleanUp` не восстанавливает `isProcessing` при ошибках до CleanUp (строка 20) — **Medium**
- **Файл:** [`src/sheets/Sheet1_main.cls`](src/sheets/Sheet1_main.cls:20)
- **Описание:** Если ошибка возникнет на строках 22-26 (до `CleanUp`), обработчик `CleanUp` сбросит `isProcessing`. Это корректно. Но если ошибка возникнет на строке 18 (`isProcessing = True`), то `CleanUp` не выполнится, и `isProcessing` останется `True`, заблокировав все последующие изменения на листе до перезапуска Excel.
- **Рекомендация:** Использовать конструкцию `On Error GoTo CleanUp` сразу после `Option Explicit` или в начале процедуры, чтобы гарантировать выполнение `CleanUp` при любой ошибке.

### 8. Mod_Logger.bas

#### LG-01: `RotateLogIfNeeded` — файл может быть не закрыт при ошибке (строка 90-91) — **High**
- **Файл:** [`src/modules/Mod_Logger.bas`](src/modules/Mod_Logger.bas:90)
- **Описание:** Файл открывается на строке 90 (`Open LogPath For Input As #F`). Если `LOF(F)` > maxSize, файл закрывается на строке 92. Если `LOF(F)` <= maxSize, файл закрывается на строке 107. **НО:** если между строками 90 и 92 возникнет ошибка (например, файл заблокирован другим процессом), файл останется открытым, что приведёт к утечке файлового дескриптора.
- **Рекомендация:** Использовать конструкцию `On Error GoTo ErrHandler` с закрытием файла в обработчике, либо использовать `FileSystemObject` для проверки размера.

#### LG-02: `WriteLog` — отсутствует проверка успешности открытия файла (строка 49) — **Medium**
- **Файл:** [`src/modules/Mod_Logger.bas`](src/modules/Mod_Logger.bas:49)
- **Описание:** `Open LogPath For Append As #F` может неявно создать файл, если его нет. Это корректно. Но если путь к файлу недоступен (например, книга открыта с сетевого пути без прав записи), ошибка будет перехвачена `ErrHandler`, который ничего не делает. Лог-сообщение будет потеряно без уведомления.
- **Рекомендация:** В `ErrHandler` можно попробовать записать в альтернативное место (например, `%TEMP%`) или хотя бы в `Debug.Print`.

#### LG-03: `ClearLog` — `Kill LogPath` может вызвать ошибку при заблокированном файле (строка 130) — **Low**
- **Файл:** [`src/modules/Mod_Logger.bas`](src/modules/Mod_Logger.bas:130)
- **Описание:** `Kill LogPath` удаляет файл. Если файл открыт другим процессом, возникнет ошибка, которая будет перехвачена `ErrHandler`. Однако после ошибки новый файл не будет создан (строки 134-136 не выполнятся), и лог останется недоступным.
- **Рекомендация:** В `ErrHandler` также создавать новый пустой файл, либо проверять доступность файла перед удалением.
